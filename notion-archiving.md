# 股票交易紀錄 歸檔機制設計

**版本：** 2.0.0
**最後更新：** 2026-06-25
**關聯文件：** [notion.md](notion.md)

---

## 1. 背景：運算量從哪裡來

效能瓶頸不在資料庫總筆數，而在 **股票交易紀錄** 的兩個公式：

| 公式 | 掃描範圍 |
| --- | --- |
| 當沖股數 | 同股票、**同日**的所有交易 |
| FIFO剩餘股數 | 同股票的**所有先前買入** + **所有賣出** |

對單一股票若有 M 筆交易，每筆是 O(M)、共 M 筆 → **O(M²)**。
真正會爆的是「**單一股票留在主表的交易鏈長度**」，所以歸檔目標是：**只把「目前還握著的批次」留在主表，其餘搬走。**

---

## 2. 依賴分析：哪些計算需要完整歷史

把 **股票代號** 會讀交易明細的公式分兩類：

| 公式 | 依賴範圍 | 搬走舊資料後 |
| --- | --- | --- |
| 持倉股數、持倉成本、持倉均價、持倉年化率 | 只看 `FIFO剩餘股數 > 0` 的買入（仍在庫存的批次） | **不受影響** |
| 淨收付合計 | `Σ 收付金額`（**全部**，含已實現） | ⚠️ 需補回 |
| 總報酬率（分母 buy_cost） | `Σ 買入收付金額`（**全部**） | ⚠️ 需補回 |
| FIFO剩餘股數（留存記錄） | 依賴被搬走記錄的「淨庫存差」 | ⚠️ 需補回 |

> 持倉相關公式天生只看「還沒賣掉的批次」，搬走已結清的舊資料不影響它們；
> 只有三個量需要把搬走那批的數值「補回來」。

---

## 3. 核心不變量

FIFO剩餘股數 的關鍵運算式：

```
max(0, min(inventory_qty, prior_buy_cum + inventory_qty − total_sell))
```

由 FIFO 先進先出可證明兩件事：

1. **被吃光的買入（FIFO剩餘=0）必定早於任何仍持有的買入** → 對每筆留存買入而言，被搬走的買入都落在它之前，影響是**固定常數**。
2. `total_sell` 與 ID 無關 → 搬走的賣出對它的影響也是**固定常數**。

因此只要補回一個常數

```
期初FIFO結轉股數 = Σ(搬走買入 股數−當沖) − Σ(搬走賣出 股數−當沖)   （恆 ≤ 0）
```

FIFO 對「留存記錄」與「未來新增記錄」都會持續正確。

> **特例：** 若搬走的那批剛好是「持倉歸零（已結清）」的完整週期，此常數 = 0，可省略 FIFO 公式的修改。一般情況（不曾結清）則必須有此常數——這就是「搬 FIFO剩餘=0」會誤算、需要修正的根因。

---

## 4. 設計總覽（推薦架構）

核心想法：**封存 DB 也關聯回「股票代號」，三個結轉值改用 Rollup 即時計算。**

```
股票代號 Database
  ├── 股票交易紀錄   (Relation, active) ── 只放「FIFO剩餘>0 的買入」+ 近期未結清交易
  │                                       → 當沖/FIFO 等 O(M²) 公式只掃這裡
  └── 封存交易紀錄   (Relation, archive)── 放已搬走的歷史交易
                                          → 三個「期初…」Rollup 從這裡即時加總
```

**為什麼這是最佳解：**

- 一筆記錄非 active 即 archive，**永遠不可能被永久重複計算**（不像手動累加會因重點按鈕而出錯）。
- 三個結轉值是**即時 Rollup**：零人工算數、零按鈕、零旗標、零「commit 與 delete 之間的視窗」。
- 封存 DB 本身就是歷史資料庫，搬移＝歸檔，一個動作完成。
- 主表只剩開倉批次，`當沖股數`／`FIFO剩餘股數` 的 O(M²) 大幅縮小。

唯一限制：Notion **無法自動跨 DB 搬頁**，搬移是手動（或用既有 updater 腳本）。這是無法避免的，但只是「移動」，不涉及任何算數或旗標維護。

---

## 5. 封存 DB 結構（股票交易紀錄_封存）

只需保存「重算結轉值所需」的欄位，且**不放 O(M²) 公式**，讓封存 DB 永遠輕量。

| 欄位 | 類型 | 說明 |
| --- | --- | --- |
| 紀錄 | Title | 沿用原命名即可 |
| 股票代號 | Relation → 股票代號 | 透過 `封存交易紀錄` 關聯回去 |
| 交易 | Select | 同主表 |
| 股數 | Number | 同主表 |
| 成交日期 | Date | 同主表 |
| 當沖股數 | **Number（凍結值）** | 搬移時把主表算好的數值複製成靜態數字，避免封存 DB 重算 |
| 收付金額 | **Number（凍結值）** | 同上，複製靜態數字 |
| 結轉貢獻FIFO | Formula | 見下，供 Rollup 加總 |
| 結轉買入收付 | Formula | 見下，供 Rollup 加總 |

> **凍結 `當沖股數`、`收付金額`** 是關鍵：搬走的是「已結清整天」，這兩個值不會再變，存成靜態數字後封存 DB 不需要任何跨筆 rollup，永遠 O(N) 輕量。

**封存 DB → 結轉貢獻FIFO：**

```jsx
(if(contains(prop("交易"), "買"), prop("股數") - prop("當沖股數"), 0))
- (if(contains(prop("交易"), "賣"), prop("股數") - prop("當沖股數"), 0))
```

**封存 DB → 結轉買入收付：**

```jsx
if(contains(prop("交易"), "買"), prop("收付金額"), 0)
```

## 6. 「股票代號」新增欄位

| 欄位 | 類型 | 定義 |
| --- | --- | --- |
| 封存交易紀錄 | Relation → 封存 DB | 第二個關聯 |
| 期初淨收付 | Rollup | 關聯 `封存交易紀錄`，Property `收付金額`，Calculate `Sum` |
| 期初買入收付 | Rollup | 關聯 `封存交易紀錄`，Property `結轉買入收付`，Calculate `Sum` |
| 期初FIFO結轉股數 | Rollup | 關聯 `封存交易紀錄`，Property `結轉貢獻FIFO`，Calculate `Sum` |

> 三者皆即時反映封存 DB 內容，無需手動維護。

---

## 7. 公式改寫（股票交易紀錄 / 股票代號）

只在原公式加入結轉項，其餘不變。

**股票交易紀錄 → FIFO剩餘股數**（新增 `carry`）：

```jsx
let(
  same, prop("交易紀錄").filter(current.prop("股票代號") == prop("股票代號")),
  carry, prop("股票代號").map(current.prop("期初FIFO結轉股數")).sum(),

  prior_buy_cum,
  same.filter(
    contains(current.prop("交易"), "買") and
    current.prop("ID") < prop("ID")
  ).map(current.prop("股數") - current.prop("當沖股數")).sum(),

  total_sell,
  same.filter(
    contains(current.prop("交易"), "賣")
  ).map(current.prop("股數") - current.prop("當沖股數")).sum(),

  inventory_qty, prop("股數") - prop("當沖股數"),

  if(
    not contains(prop("交易"), "買") or inventory_qty == 0,
    empty(),
    max(0, min(inventory_qty, prior_buy_cum + carry + inventory_qty - total_sell))
  )
)
```

**股票代號 → 淨收付合計：**

```jsx
let(
  records, prop("股票交易紀錄"),
  records.map(current.prop("收付金額")).sum()
  + if(empty(prop("期初淨收付")), 0, prop("期初淨收付"))
)
```

**股票代號 → 總報酬率**（只改 `buy_cost`）：

```jsx
if(
  prop("淨收付合計") != 0,
  let(
    buy_cost,
    prop("股票交易紀錄").filter(
      contains(current.prop("交易"), "買")
    ).map(current.prop("收付金額")).sum() * -1
    - if(empty(prop("期初買入收付")), 0, prop("期初買入收付")),
    if(buy_cost > 0, prop("總損益") / buy_cost, empty())
  ),
  empty()
)
```

> `期初買入收付` 為負數，`buy_cost` 取正成本，故為 `... - 期初買入收付`。
>
> **相容性備註：** 若你的 Notion 版本不支援在 `股票交易紀錄` 公式中經 relation 讀取 `股票代號` 的 **Rollup**，則在 `股票代號` 加一個 formula `期初FIFO結轉股數F = prop("期初FIFO結轉股數")` 鏡像該 Rollup，再於 FIFO 公式改讀 `...map(current.prop("期初FIFO結轉股數F"))`。（讀取「關聯頁的 formula」是系統既有用法，必可運作。）

## 8. `可歸檔` 判斷欄位（股票交易紀錄）

把「哪些可以搬」做成公式，搬移時直接篩 `可歸檔 = true`，從源頭避免搬錯。
條件：本筆 **FIFO剩餘 = 0/empty** **且** 同股票同日 **沒有任何 FIFO剩餘 > 0 的記錄**（§9）。

```jsx
let(
  same_day,
  prop("交易紀錄").filter(
    current.prop("股票代號") == prop("股票代號") and
    current.prop("成交日期") == prop("成交日期")
  ),
  day_has_holding,
  same_day.filter(current.prop("FIFO剩餘股數") > 0).length() > 0,
  my_fifo, prop("FIFO剩餘股數"),
  (empty(my_fifo) or my_fifo == 0) and not day_has_holding
)
```

- 回傳 `true` ＝可安全搬走。因「整天無留存」才會 true，篩出的記錄天然以**整天**為單位。
- 只讀取 `FIFO剩餘股數`（單向依賴），不形成循環。

---

## 9. 必守規則：不可切壞同一個交易日

`當沖股數` 依賴「同日的買賣總量」。若某日仍有 FIFO剩餘 > 0 的留存買入，搬走同日其他記錄會改變該留存買入的當沖計算 → FIFO 連帶出錯。

> **只搬「整天」，且該日不得有任何 FIFO剩餘 > 0 的記錄。**
> §8 的 `可歸檔` 已內建此判斷，照它篩即安全。

## 10. 完整範例

某股票（單位：股，皆現股非當沖）：

| ID | 交易 | 股數 | 收付金額 | FIFO剩餘 | 可歸檔 |
| --- | --- | --- | --- | --- | --- |
| 1 | 現買 | 1000 | −50,000 | 0 | ✓ |
| 2 | 現買 | 1000 | −52,000 | 500 | ✗（持倉中）|
| 3 | 現賣 | 1500 | +78,000 | empty | ✓ |

歸檔前：持倉股數 = 500、淨收付合計 = −24,000、買入總成本 = 102,000。

**把 ID 1、3 搬到封存 DB（關聯回此股票），主表只剩 ID 2：**

封存 DB 的三個 Rollup 自動算出：

- 期初FIFO結轉股數 = (買1000) − (賣1500) = **−500**
- 期初淨收付 = (−50,000) + (+78,000) = **+28,000**
- 期初買入收付 = (−50,000) = **−50,000**

驗證（主表只剩 ID 2）：

- FIFO剩餘(ID2) = min(1000, 0 + (−500) + 1000 − 0) = **500** ✓
- 持倉股數 = 500 ✓
- 淨收付合計 = (−52,000) + 28,000 = **−24,000** ✓
- buy_cost = 52,000 + 50,000 = **102,000** ✓

全部與歸檔前一致。

## 11. 歸檔操作流程

1. 一次性建好：封存 DB（§5）、`封存交易紀錄` Relation 與三個 Rollup（§6）、改三個公式（§7）、加 `可歸檔`（§8）。
2. 對單一股票，篩 **`可歸檔 = true`**。
3. 把這些記錄搬到封存 DB：
   - 複製 交易 / 股數 / 成交日期；
   - 把 **`當沖股數`、`收付金額` 以「數值」貼成靜態 Number**（不要帶公式）；
   - 設定 `股票代號`（封存端 Relation）指向同一檔股票。
4. 從主表刪除這些記錄。
5. **核對**該股票 持倉股數 / 持倉成本 / 淨收付合計 / 總報酬率 與步驟 2 之前一致。

> 建議用既有的 updater 腳本（Fugle/Notion API）把步驟 3–4 自動化：讀 `可歸檔=true` → 建立封存頁（含凍結值）→ 刪主表頁，可完全免手動。

---

## 12. 注意事項

1. **整檔結清的股票**：交易明細可全搬到封存，但「股票代號」那一列要**保留**（承載 Rollup 與已實現 總損益），否則 資產 Database 會少算。
2. **資產 Database 不需改**：其 Rollup 讀的是 股票代號 已修正的公式，自動正確。
3. **搬移的瞬時性**：Notion 跨 DB 搬頁若拆成「先建封存、後刪主表」，中間會有 sub-second 的暫態（短暫多算或少算），完成後即自動正確；因為值全是即時 Rollup，**不可能產生永久性的重複計算**。
4. **凍結值**：封存 DB 的 `當沖股數`、`收付金額` 必須是靜態 Number；若誤設為公式且封存 DB 缺少同股票 rollup，會算錯。
5. **稅率三處**：本機制不動 賣出費用 / 損益平衡價 / 總損益平衡價，原規則不變。

## 附錄：純手動 fallback（不建封存 Relation）

若不想建第二個 Relation，可改用「`股票代號` 上的三個手動 Number 欄位 + Button 累加」：公式相同，但 `期初…` 改為手動值。**缺點**：需人工算 Σ、按鈕重複點會重複加、commit 與 delete 間有暫態——故僅作為簡化備案，正式建議採用 §4 的 Rollup 架構。
